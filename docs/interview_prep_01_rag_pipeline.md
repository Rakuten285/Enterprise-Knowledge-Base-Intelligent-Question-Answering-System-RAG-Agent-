# 面试准备文档：RAG 核心链路

> 本文档与代码同步维护。目标不是背答案，而是讲清楚"为什么这么设计"。
> 每个问题先讲结论，再讲推理过程——面试官通常更想看你的思考路径，而不是标准答案本身。

---

## 1. 整体架构：为什么这样分层？

**问：你的 RAG 链路是怎么组织代码的？**

```
文档加载(loaders) → 分块(splitter) → Embedding(embeddings) → 向量库(vectorstore)
                                                                      ↓
                                                          BM25(retrievers) → 混合检索融合 → MMR去重
```

每一层独立成模块，靠**接口而非具体实现**互相依赖：
- `BaseEmbeddingProvider` 是抽象基类，`bge_local` / `openai` / `zhipu` 三个实现都遵守同一接口（`embed_documents` / `embed_query` / `dimension`）
- `FAISSVectorStore` 和 `BM25Retriever` 互不知道对方存在，靠 `HybridRetriever` 组合
- 工厂函数（`get_embedding_provider` / `get_llm`）统一做实例化，业务代码不直接 `import` 具体 Provider 类

**为什么这么做**：换 Embedding 模型（比如从本地 BGE 换成 OpenAI）只需要改 `.env` 里的 `EMBEDDING_PROVIDER`，不用动任何检索、分块、向量库的代码。这是依赖倒置原则的直接应用——上层模块依赖抽象，不依赖具体实现。

**可能被追问**：如果面试官问"如果要新增一个 Embedding Provider（比如百度文心）怎么加？"——答案是：在 `app/rag/embeddings/` 下新建一个文件，继承 `BaseEmbeddingProvider`，实现三个抽象方法，再在 `factory.py` 的 `get_embedding_provider` 里加一个分支即可，**不需要改动 `FAISSVectorStore`、`HybridRetriever` 等任何下游代码**。这正是开闭原则的体现。

## 2. 文档解析与分块

### 2.1 为什么 PDF 要逐页解析，Word 不用？

**问：load_pdf 和 load_docx 的实现思路不一样，为什么？**

PDF 按页拆分成多个 `Document`，每个带 `page` 元数据；Word 整篇拼成一个 `Document`。

原因是 **PDF 天然有"页"这个物理边界，且页码对溯源很有价值**（用户问"这是手册第几页写的"，能直接答上来）；而 Word 文档的分段更多依赖标题层级和段落，页码在 docx 文件结构里不是一等公民信息（python-docx 读不到"渲染后第几页"，只能读段落顺序），所以不强求按页，而是先整合成完整文本，交给后面统一的分块器处理。

表格内容我做了单独提取（遍历 `table.rows`，每行用 `|` 拼接），附加在正文后面用 `[表格内容]` 标记。**为什么不混在正文里**：表格是结构化的行列数据，如果不加标记直接拼进段落文本，分块时容易把一行表格切断在两个 chunk 之间，导致检索时语义破碎（比如"5000元以下"和"7个工作日内"被分到不同块）。加标记至少让后续做表格专用解析（比如转 markdown 表格）留了改造空间。

### 2.2 为什么用 RecursiveCharacterTextSplitter，分隔符为什么这么设计？

**问：分块策略是怎么选的？**

`RecursiveCharacterTextSplitter` 的核心逻辑是**按分隔符优先级递归尝试切分**：先试最高优先级的分隔符（比如段落分隔 `\n\n`），如果切出来的块还是超过 `chunk_size`，就换下一级分隔符（比如句号）继续切，直到块大小达标或分隔符用完（退化到按字符硬切）。

我把默认分隔符列表换成了中文优先级：

```python
["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""]
```

**为什么不用默认的英文分隔符**（默认是 `["\n\n", "\n", " ", ""]`）：英文默认列表没有中文句号、问号这些标点，会导致中文长段落只能按 `\n` 或硬切，容易把一句话切断在中间，破坏语义完整性。把中文标点按"段落 > 句子 > 子句 > 词"的粒度排进去，能让切分点尽量落在语义边界上。

**chunk_size / chunk_overlap 怎么选的**：默认 500/50，这是评测阶段要调的参数（这部分会在 Ragas 评测里详细讲，目前是先给一个工程经验值占位）。`overlap` 存在的意义是避免关键信息恰好卡在两个 chunk 的边界被切断、两边都拿不到完整上下文。

### 2.3 chunk_index 和 total_chunks 这两个 metadata 字段是干什么的？

分块之后，我按 `source` 文件名重新分组编号（不是用 LangChain splitter 自带的全局序号），这样每个 chunk 知道自己是"这篇文档的第几块/共几块"。

**为什么要这个**：检索结果展示给用户时，能告诉他"这是《员工手册》第 3/12 块"，比单纯一个全局编号更有可读性，也方便后续做"取相邻 chunk 补充上下文"这种优化（比如检索到第3块发现信息不完整，可以联动取第2、4块）。当前版本还没实现这个联动，但 metadata 结构已经为它留了位置。

## 3. Embedding 选型

### 3.1 为什么默认用本地 BGE，不是 OpenAI？

**问：项目描述里写的是 text-embedding-3-small，但你代码默认用 BGE，为什么？**

诚实回答这个问题很重要——这恰恰是个好的展示点：**原始需求文档（可能来自之前的项目经验总结或参考案例）用的是 OpenAI，但实际工程决策要考虑可复现性、成本、合规性**。OpenAI Embedding 需要国外 API Key 和网络环境，对于一个要开源给别人直接 `git clone` 跑起来的项目，这个门槛太高，会变成"代码能跑但大多数人跑不起来"。

所以我做了 Provider 抽象层，**默认用 `BAAI/bge-small-zh-v1.5`**：完全免费、离线运行、无需注册任何账号，中文检索效果在中文榜单（C-MTEB）上也不输 OpenAI 同级模型。同时保留 OpenAI / 智谱两个可选项，三选一通过环境变量切换——这样既保留了"对标 text-embedding-3-small"的技术对比能力，又让项目本身门槛降到最低。

**可能被追问：BGE 和 OpenAI Embedding 的核心区别是什么？**
- 维度不同：`bge-small-zh` 是 512 维，`text-embedding-3-small` 是 1536 维。维度越高理论上表达能力越强，但对于中文场景，bge 系列经过中文语料专门训练，中文语义捕捉不一定弱于通用大模型的 embedding。
- BGE 官方推荐**查询侧加指令前缀**（"为这个句子生成表示以用于检索相关文章："），文档侧不加——这是非对称检索（asymmetric retrieval）的常见技巧，因为 query 通常短、文档通常长，分别处理能提升匹配效果。这点我在 `bge_local.py` 的 `embed_query` 里专门做了区分，`embed_documents` 没加前缀。
- 归一化：调用时传了 `normalize_embeddings=True`，这样向量点积直接等价于 cosine 相似度，FAISS 用内积索引时不用再单独算 cosine。

### 3.2 `@lru_cache` 用在 Embedding/LLM 工厂函数上是为什么？

模型加载（尤其本地 BGE）有实际开销——读取几百 MB 的模型权重到内存。如果每次调用 `get_embedding_provider()` 都重新实例化，会反复触发模型加载，拖慢请求。用 `lru_cache` 把工厂函数变成单例模式：同样的参数只会真正执行一次模型加载，后续调用直接返回缓存的实例。

**注意**：`lru_cache` 默认是按参数做 key 缓存的，所以 `get_llm(provider="deepseek", temperature=0.1)` 和 `get_llm(provider="deepseek", temperature=0.7)` 会被当成两个不同的缓存条目，分别实例化——这是合理的，因为 temperature 不同确实需要不同的模型实例。

## 4. 混合检索：这是整个项目最容易被深挖的部分

### 4.1 为什么向量检索不够，要加 BM25？

**问：向量检索已经能做语义匹配了，为什么还要用 BM25？**

向量检索和关键词检索是互补的，不是替代关系：

- **向量检索的强项**：语义相似、同义改写、跨表达方式的匹配。比如用户问"离职要提前多久说"，即使原文写的是"离职需提前30天...提交申请"，向量也能捕捉到语义相关性。
- **向量检索的弱项**：对**精确字面匹配**（专有名词、编号、数值、系统名称）反而不一定敏感。比如用户搜"IT-开头的工单编号怎么用"，或者搜"SecureLink"这个具体产品名，向量模型如果没在训练数据里见过这种生僻专名，可能无法给出高相似度匹配；但 BM25 是基于词频统计的，只要分词后字面命中，分数就会很高。

我专门在示例文档里放了一些这种场景（比如《IT安全规范》里的 "SecureLink"、"NextCloud"、"IT-"开头的工单编号），就是为了在后续评测阶段能实际验证这个优势，而不是空口说"提升15%"。

**追问：15% 这个提升数字怎么来的？**——如实说：这是规划阶段的目标值，需要后续用 Ragas 在 50 条问答对评测集上对比"纯向量检索"和"混合检索"两个版本的 Context Recall 实际跑出来，不是凭空编的。如果面试时还没跑出实测数据，要诚实说"这是预期目标，实际数字以评测结果为准"，千万不要编一个数字说死。

### 4.2 RRF 和加权融合，两种策略的本质区别是什么？

**这是这个模块的核心考点，务必讲清楚。**

**加权融合（weighted）**的问题在于：向量检索返回的是 **L2 距离**（欧式距离，越小越相似），BM25 返回的是**词频统计分数**（TF-IDF变体，越大越相关）。这两个分数**量纲完全不同**——一个可能在 0~2 之间，另一个可能是 0~50，直接加权求和毫无意义，必须先做 min-max 归一化把两边都缩放到 [0,1] 再加权。

但归一化本身有个问题：**它对分数分布很敏感**。如果某次检索里 BM25 所有候选分数都很接近（比如查询词很常见，大家命中次数差不多），归一化后差异会被放大或压平，可能导致排序不稳定。

**RRF（Reciprocal Rank Fusion）**的思路不一样：**完全不看原始分数，只看排名**。公式是：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

对每个文档，把它在向量检索里的排名和在 BM25 里的排名分别转换成 `1/(k+rank)`，加起来。`k` 是个平滑常数（我用了业界常见的 60），排名越靠前（rank 越小），贡献分越高；`k` 越大，排名差异的影响就越平滑。

**为什么 RRF 更鲁棒**：它天然规避了"两路分数量纲不一致"的问题，因为排名是无量纲的——不管 BM25 的分数是 3 还是 300，只要它排第一，对融合分的贡献就是固定的 `1/(k+1)`。这也是为什么我把 `rrf` 设成默认策略，`weighted` 作为可选项保留（用于做对比实验，或者某些场景下你确实想人工控制两路检索的权重偏向）。

**追问：k=60 这个值哪来的？**——这是 RRF 原始论文（Cormack et al., 2009）和后续很多工业实践中的常见经验值，不是我调出来的最优解。如果追问"你试过别的 k 值吗"，诚实回答：这是工程默认值，留了 `RRF_K` 环境变量可调，严谨做法应该是像 chunk_size 一样在评测集上做参数扫描，目前还没做这一步（如果做了评测后验证过，到时候更新这里）。

### 4.3 候选池大小（fetch_k）和最终返回数（k）为什么要分开？

检索流程是：**向量召回 N 个候选 + BM25 召回 N 个候选 → 融合 → 候选池截断 → （可选）MMR 重排 → 最终返回 k 个**。

这里 N（候选池大小，对应 `mmr_fetch_k`）通常比最终的 k 大很多（比如 k=5，候选池=20）。

**为什么不直接召回 k 个**：如果两路检索各自只取 top-5，融合时可能会丢失"在某一路排名靠后但实际相关"的文档——比如某个文档在向量检索里排第 8（因为语义相似度算法没那么敏感），但在 BM25 里排第 1（因为精确命中关键词），如果向量那边只取了 top-5，这个文档在融合阶段就完全看不到了。扩大候选池是为了**给融合算法足够的"原始材料"去做正确判断**，这是混合检索能生效的前提条件，不是可选的优化项。

### 4.4 MMR 是在哪个阶段做的？为什么不在融合之前做？

**这是个容易被问到设计取舍的点。**

我的 MMR 实现是放在**融合之后**——先把向量+BM25的结果融合成一个统一候选列表，再对这个**融合后的列表**做 MMR 去重，而不是分别在向量检索结果、BM25结果上各自做一次 MMR。

**为什么这样设计**：MMR 的目的是"减少返回结果之间的冗余"，这个目标应该作用在**最终展示给用户/喂给LLM的结果集**上，而不是中间过程。如果在融合前的两路结果上分别做MMR，反而可能在融合阶段又把冗余的内容从两路里重新拼回来（比如向量检索去重后剩下A、C，BM25去重后剩下B、D，融合后A和B如果语义高度相似，冗余问题又出现了）。所以"先融合、后MMR"在逻辑上更彻底。

**MMR 的实现细节**：我没有依赖 FAISS 自带的 `max_marginal_relevance_search`（虽然 `faiss_store.py` 里也保留了这个方法，可以单独在纯向量检索场景下用），因为混合检索融合之后的候选集已经不是单纯的 FAISS 检索结果了，没法直接复用 FAISS 内部的 MMR 实现。我用 numpy 手写了一个等价的贪心算法：

```
relevance(d) = cosine_sim(query, d)
MMR_score(d) = λ * relevance(d) - (1-λ) * max(cosine_sim(d, already_selected))
```

每一步贪心选择 MMR_score 最高的候选加入结果集，直到选够 k 个。`λ` 越接近 1，越偏向纯相关性（容忍冗余）；越接近 0，越强调多样性（牺牲一些相关性去换取信息覆盖面）。

**追问：MMR 的时间复杂度，候选池一大会不会很慢？**——诚实分析：每选一个新文档，要和已选集合里每个文档算一次 cosine 相似度，整体是 O(k × pool_size) 次相似度计算，外加要先对候选池里所有文档重新计算一次 embedding（如果候选文档之前没缓存向量）。当前实现里 `_apply_mmr` 确实会对候选文档**重新调用 `embed_documents`**——这是一个已知的效率问题，没有复用建库时已经算好的向量，候选池稍大（比如几十到上百）时会有明显的重复计算开销。更优化的做法是在建库阶段把每个 chunk 的向量缓存下来（比如存在 FAISS 之外再存一份 id→vector 的映射），MMR 阶段直接查表而不是重新 embed。**这是当前实现的一个已知短板，可以在面试里主动提出来作为"如果有更多时间会怎么优化"的回答**，比一问三不知要好得多。

### 4.5 `_doc_key` 用 source + chunk_index 做去重键，会不会有问题？

会有一个边界情况：如果同一个文件被**重复建库两次**（比如脚本跑了两遍没清理旧索引），可能出现 source 和 chunk_index 都相同但内容不同的情况（如果两次建库用了不同的 chunk_size）。当前实现没有处理这种"索引版本冲突"问题，正常使用流程（建库前清空旧索引或走增量更新接口）不会触发，但如果被问到"会不会有重复数据"的边界情况，要能说出这一点。

## 5. BM25 实现细节

### 5.1 为什么要自己写停用词表，不用现成的库？

`rank_bm25` 库本身只管分数计算，不包含中文分词和停用词处理，这两步需要自己做。我用 `jieba.lcut` 分词，再过滤掉一个手写的停用词集合（"的"、"了"、"是"等高频虚词）。

**为什么停用词很重要**：BM25 本质是基于词频统计的，如果不过滤虚词，"的"、"了"这种几乎在每个文档里都高频出现的词会拉低真正关键词的区分度（虽然 BM25 公式里的 IDF 项本身会压低高频词的权重，但中文虚词数量大、过滤掉能让计算更高效，且部分边界情况下IDF压制不够彻底）。

**这个停用词表目前是手写的小集合，不是工业级词表**——如果追问"够用吗"，诚实回答：这是一个验证可行性的最小集合，生产场景应该用更完整的停用词库（比如哈工大停用词表），这是可以快速替换的一行配置，不是架构问题。

### 5.2 BM25 索引为什么要持久化（pickle），不能每次重新分词建索引吗？

`rank_bm25.BM25Okapi` 初始化时需要传入**已分词的语料**（`tokenized_corpus`），如果文档量大，每次启动服务都重新对全量文档分词、重新计算 IDF 统计量，会有明显的启动延迟。所以我把分词后的语料和原始 Document 列表一起 pickle 序列化落盘，加载时直接反序列化重建 `BM25Okapi` 对象，跳过重复分词这一步。

**注意**：这里持久化的是"分词后的语料"，不是 BM25Okapi 对象本身直接 pickle（虽然技术上也可以直接 pickle 整个对象）。这样做的好处是如果以后升级了 `rank_bm25` 库版本、或者想换一种 BM25 变体（比如 BM25L、BM25+），可以直接复用已经分好词的语料重新构建，不需要重新分词。

## 6. RAG 生成链（chain.py）：检索结果如何转化为最终答案

> **阶段1最后一块**，2026-06-18 已实测跑通，以下是真实运行结论，不是预期描述。

### 6.1 整体架构

```
用户 query
    ↓
pipeline.retrieve()        ← 混合检索，返回 top-k RetrievedChunk 列表
    ↓
chain.generate_answer()    ← Prompt 拼装 + LLM 调用，返回 RAGAnswer
    ↓
{ answer, sources, retrieved_chunks }
```

`chain.py` 只做一件事：把 `RetrievedChunk` 列表和 query 拼成 Prompt，调 LLM，拿回文本。它不做检索，不关心 embedding，职责非常单一。

### 6.2 Prompt 设计：为什么这么写

System Prompt 核心是两条约束：

```
1. 只基于给定的参考文档作答，不要使用文档之外的知识
2. 如果参考文档中没有足够的信息，明确告知用户"文档中未找到相关信息"，不要编造
```

**为什么这两条最重要**：RAG 系统最大的风险不是"答不上来"，而是"用模型自身的训练知识编出听起来合理但和公司实际规定不符的答案"。比如用户问年假天数，如果模型没有被约束，可能直接说"国家法定年假是5天"（这是通用知识），而不是基于公司文档里的实际规定。用 System Prompt 强制约束"只读文档"，是 RAG 系统的基本安全线。

**追问：System Prompt 能真的 100% 防止幻觉吗？**
不能。约束 Prompt 能显著降低但不能完全消除幻觉，特别是当检索到的文档和问题"相关但不够精准"时，模型可能用文档里的内容做不正确的推断。真正可靠的防幻觉需要多层手段：①好的检索（确保相关文档能被召回）+②约束 Prompt +③后处理置信度评分（比如 Ragas 的 Faithfulness 指标）。这三层都是这个项目要覆盖的，当前只做了前两层。

### 6.3 Context 拼装格式

```
[1] 来源：员工手册.md
<chunk 内容>

[2] 来源：IT安全规范.md
<chunk 内容>
...

---
用户问题：公司年假有多少天
```

**为什么给每个 chunk 编号和来源**：
- 编号让 LLM 可以在回答里引用（"根据第[1]条文档..."），也方便调试时对比"LLM引用了哪条"和"检索返回了哪条"
- 来源文件名是 LLM 做"根据《员工手册》..."这类引用表述的依据，不给来源 LLM 就无法做有意义的出处标注

### 6.4 实测结果（3条测试，2026-06-18）

**测试1：知识库内有明确答案**
- 问题：`公司年假有多少天`
- 回答：分三档（1-10年5天、10-20年10天、20年以上15天），注明来源《员工手册》
- 结论：✅ 正确提取多段信息并结构化输出

**测试2：精确字面信息（密码规则）**
- 问题：`密码长度要求是多少，连续输错几次会锁定`
- 回答：不少于12位，连续5次锁定，注明来源《IT安全规范》
- 结论：✅ 正确，说明 BM25 精确字面匹配在这类"关键数字"问题上确实有效

**测试3：知识库外的问题（反幻觉验证）**
- 问题：`员工去世后家属能拿到哪些补贴`
- 回答：`未在知识库中检索到与您问题相关的文档内容`，建议咨询人力资源部
- 结论：✅ 正确拒答，没有用通用知识编造，System Prompt 约束生效

**面试素材**：第3条测试结果是最有价值的——RAG 的价值不只是"能答上来"，而是"答不上来时能正确说不知道"。很多 LLM 系统出问题恰恰在这里：问的问题文档里没有，模型用通用知识拼凑了一个"感觉像对的"答案，用户信了，结果出错。主动设计并验证这个"拒答"场景，说明你对 RAG 系统的可靠性边界有清晰认知。

### 6.5 `RAGAnswer` 数据结构设计

```python
@dataclass
class RAGAnswer:
    answer: str                          # 生成的文字回答
    sources: list[str]                   # 去重后的来源文件名列表
    retrieved_chunks: list[RetrievedChunk]  # 原始检索片段（供调试/评测复用）
```

`retrieved_chunks` 保留下来的原因：后续阶段做 Ragas 评测时，需要同时拿到"生成的答案"和"检索到的上下文"才能计算 Faithfulness（答案是否忠于检索结果）和 Context Recall（检索结果是否覆盖了正确答案所需的上下文）。如果生成时把原始 chunks 丢掉，评测阶段就得重跑一次检索，既低效又可能因为模型非确定性导致两次结果不一致。把它塞进返回结构，零成本保留了评测所需的所有信息。
